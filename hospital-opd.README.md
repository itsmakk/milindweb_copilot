# Hospital OPD SPA — MilindWeb

A zero-dependency, framework-free OPD registration SPA that replaces the
legacy `form.html` (which used a Google Apps Script backend).

## Quick start

1. Sign in at `/auth/login.html` with an account in one of the
   `milindweb-hospital-*` Authentik groups (reception, doctor, admin, …).
2. Open `/opd.html`. The form auto-loads the active doctors + departments
   from the API.
3. Search for a patient by name / mobile / UHID, or click "New patient"
   to register one (UHID is auto-generated server-side as `MH-YYYY-NNNNNN`).
4. Fill the visit form, add prescriptions, save. The OPD number
   (`OPD-YYYY-MM-DD-NNNN`) and UHID are generated by the API.
5. The confirmation step lets you print a visit slip on a clean A4 layout
   (uses `css/print.css`).

## Files

```
opd.html         # the SPA shell
js/opd.js        # auth, search, save, prescriptions, autocomplete
css/opd.css      # clean modern styling (replaces the old flat gov style)
form.html        # now a thin redirector (preserves the old URL)
```

## How it works

- **Auth guard** checks `MW_AUTH.getSession()` and calls `/auth/me` to
  resolve the role. Hospital staff only — anyone else sees a "forbidden"
  screen with a link back to `/auth/me.html`.
- **Reference data** (doctors, departments) is loaded once at boot from
  `/hospital/doctors?active=true` and `/hospital/departments?active=true`.
- **Symptom autocomplete** uses `data/dept.json` (the same JSON the old
  form used) as a fallback. DB-backed `Department.defaultSymptoms` is
  preferred when available.
- **Medicine autocomplete** uses `data/medlist.json` (the same JSON the
  old form used). Drug DB integration is a future phase.
- **Prescriptions** are sent to `/hospital/visits/:id/prescriptions` one
  at a time (the API exposes a per-RX endpoint; a batch endpoint is a
  future optimisation).
- **Print** uses `css/print.css` which `app.js` auto-injects into every
  page. The visit slip hides the header/footer/nav and shows a clean
  `.mw-print-header` with the OPD number + timestamp.

## Roles

| Action                                          | Required role(s)               |
|-------------------------------------------------|--------------------------------|
| Read departments / doctors / search patients    | any `hospital_*`               |
| Create patient                                  | `hospital_admin`, `hospital_reception`, `hospital_doctor` |
| Create visit (consultation)                     | `hospital_admin`, `hospital_doctor`, `hospital_reception`, `hospital_nurse` |
| Edit clinical fields (provisional/final Dx, Rx) | `hospital_admin`, `hospital_doctor` |
| Add prescription                                | `hospital_admin`, `hospital_doctor` |
| Soft-delete any record                          | `hospital_admin`               |

The role checks are enforced server-side; the SPA shows/hides buttons
accordingly, but the API is the source of truth.

## Limitations

- **No file upload.** Signature image, lab reports, photos: defer to a
  later phase (Storage module).
- **No batch prescription endpoint.** Each Rx is a separate POST. Fine
  for visits with < 20 prescriptions; add a `POST /visits/:id/prescriptions/batch`
  if it ever matters.
- **Symptom + medicine autocomplete come from static JSON files.** When
  the DB has enough rows, switch them to query `/hospital/symptoms?` and
  `/hospital/medicines?` (future endpoints).
- **No edit-after-save in the SPA.** Once a visit is saved, you can only
  print or start a new one. Editing is via `/hospital/visits/:id` (PATCH)
  from a future "visit history" page.
